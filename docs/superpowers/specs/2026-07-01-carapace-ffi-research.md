# carapace-ffi — cross-platform embedding research (2026-07-01)

Deep-research pass (105 agents, adversarially verified) feeding the `carapace-ffi` design: how to
promote the throwaway `embed-spike` prototype into a **stable, safe, cross-platform** FFI/embedding
layer that lets host apps embed the wgpu+Vello engine as custom UI chrome. macOS/iOS are already
proven this session; the focus is a durable ABI + threading model + a per-platform texture-sharing
plan, with the **Windows and Linux unknowns** called out. Findings are high-confidence unless noted;
`(vote x-y)` is the adversarial verification tally.

## Bottom line / recommended architecture

- **ABI:** a versioned C ABI over **opaque handles**, header generated by `cbindgen` (or the
  `abi_stable`/`stabby` stable-ABI crates if we ship a plugin-style dynamic boundary). Any Rust type
  crossing the boundary MUST be explicitly `repr(C)` or a stable-ABI type — `repr(Rust)` layout is
  deliberately unstable.
- **Panic safety (non-negotiable):** every exported entry point is **abort-on-panic**. Either
  `extern "C-unwind"` (stable since Rust 1.71) with an internal `catch_unwind` returning error
  codes/null, or `abi_stable`'s `sabi_extern_fn` / `extern_fn_panic_handling` which insert an abort
  guard. A panic unwinding into foreign frames is UB; this is exactly the `init_gpu().expect()` hole
  we flagged.
- **Threading:** a **dedicated render thread owns the `!Send`/`Rc` engine**, driven by a command
  queue; the C ABI marshals calls onto it. This matches host contracts (e.g. Flutter registers
  textures on the platform thread but consumes frames on the raster thread) and fixes the
  synchronous main-thread readback lag both the macOS and Flutter spikes hit.
- **Texture sharing:** all zero-copy paths run through **`wgpu-hal` unsafe escape hatches**, not
  wgpu's safe public API. Apple is proven; Windows and Linux have concrete but low-level, churn-prone
  HAL paths. Build against `wgpu-hal` and budget for its instability.

## (1) Stable C ABI + panic/unwind safety

- **Abort-on-panic at the boundary.** Under `panic=unwind`, a panic escaping a plain `extern "C"` fn
  aborts (guaranteed, not UB, since Rust 1.81); `extern "C-unwind"` (RFC 2945, stabilized 1.71)
  *permits* unwinding both directions — so we must *deliberately* choose abort + internal
  `catch_unwind` → error code/null. (vote 3-0) — RFC 2945; Rust Reference `panic.html`.
- **Ready-made discipline:** `abi_stable`'s `sabi_extern_fn` defines `extern "C"` fns that **abort on
  unwind instead of UB**, and `extern_fn_panic_handling` inserts an `AbortBomb` guard. (3-0) — docs.rs/abi_stable.
- **Versioning/compat:** `abi_stable` does **load-time layout verification** + semver-safe extension
  (**Prefix Types** for extensible vtables/modules, **`NonExhaustive` enums** for adding variants) —
  preferable to hand-rolled versioning *if* we ship a dynamic plugin boundary. (3-0) — docs.rs/abi_stable.
- **`stabby`** is a lighter alternative (per-type/fn annotation, deterministic layout via `IStable`).
  Note: `#[stabby::stabby]` is **not** plain `repr(C)` passthrough — it layers `repr(C)` + niche
  optimizations (that sub-claim was **refuted 1-2**). (3-0) — github.com/ZettaScaleLabs/stabby.
- Not yet evaluated: `abi_stable`/`stabby` vs plain `cbindgen` **head-to-head for carapace's specific
  needs** (a simple flat C ABI probably wants cbindgen; a versioned plugin boundary wants abi_stable).

## (2) Threading a single-threaded engine over FFI

- Dedicated render thread + command queue is the right shape; host toolkits already impose
  thread-affinity that a single "call the engine from anywhere" ABI would violate. Flutter is the
  concrete example: **register/unregister textures on the platform thread**, but frames are consumed
  on the **raster thread** via `copyPixelBuffer`. (3-0) — Flutter iOS embedder docs.
- Open: the exact triple-buffering / acquire-release-barrier scheme to minimize present latency while
  reconciling those differing per-host affinities (see Open Questions).

## (3) Cross-platform zero-copy GPU texture sharing

**The wgpu reality (critical):** wgpu's **safe/public API has no external-texture import**. The
formal interop framework (**issue #4067**, open since Aug 2023) and the external-memory import
request (**#2320**, open since Dec 2021) are **design intent, not shipped APIs**. Zero-copy import is
only reachable through **`wgpu-hal`** — carapace-ffi must build against wgpu-hal, not just wgpu.
(3-0 / 2-1) — wgpu #2320, #4067.

- **The import entry point:** `Device::create_texture_from_hal` wraps a foreign backend texture and
  now takes an explicit `initial_state: wgt::TextureUses` (declares the state the wrapped resource is
  already in). Backend `texture_from_raw`/`texture_from_*` calls create the HAL texture from platform
  handles. (3-0) — wgpu trunk CHANGELOG.
- **Lifetime/sync contract:** a **`DropGuard` (`Box<dyn FnOnce>`)** decides ownership — *with* a
  guard, the host owns the resource and wgpu will **not** destroy it (host keeps it alive until the
  guard fires); *without*, wgpu takes ownership and destroys it. This is the key contract when the
  host owns the shared surface. (Caveat #6142: drop-guard semantics are inconsistent across some
  `from_raw` variants.) (3-0) — wgpu #4067, PR #3355.

- **Apple — PROVEN, shippable.** IOSurface-backed `MTLTexture`
  (`newTextureWithDescriptor:iosurface:plane:`, shares memory with the IOSurface, no copy) →
  `wgpu::hal::metal::Device::texture_from_raw`. Exactly what **Slint+Servo ship** and carapace's
  proven macOS/iOS route. (3-0) — slint.dev/blog/using-servo-with-slint.
- **Windows — concrete HAL path (D3D11→Vulkan).** The Vulkan backend imports a **D3D11 shared handle**
  as a Vulkan texture via **`texture_from_d3d11_shared_handle`** using `VK_KHR_external_memory_win32`
  (**merged Oct 30 2024**, PR #6161; handle from `IDXGIResource1::CreateSharedHandle`, type
  `…D3D11_TEXTURE_BIT`). Direction is D3D11→Vulkan import; caller must `CloseHandle` the NT handle;
  hal-level, not public API. **The prior claim that "Windows has no zero-copy path" was REFUTED 0-3 —
  do not treat Windows as unworkable.** (3-0) — wgpu PR #6161.
- **Linux — HAL path (dmabuf), trunk-only.** `vulkan::Device::texture_from_dmabuf_fd()` imports
  DMA-buf textures, gated on `VULKAN_EXTERNAL_MEMORY_FD` / `VULKAN_EXTERNAL_MEMORY_DMA_BUF`. The
  GL-consumer half: export fd via `vkGetMemoryFdKHR`, import via `GL_EXT_memory_object_fd` (also
  covers Android). RISK: `texture_from_dmabuf_fd` was **trunk/Unreleased** at research time (not in a
  stable wgpu release); one CHANGELOG entry mis-cites PR #9412 (doc blemish). (3-0 / 2-1) — wgpu
  trunk source `wgpu-hal/src/vulkan/device.rs`, CHANGELOG; slint.dev blog.

## (4) Host-toolkit embedding surfaces

- **Flutter:** host implements `FlutterTexture`, `registerTexture:` → `int64_t` id (0 = failure),
  `textureFrameAvailable:` signals frames. **iOS delivers `CVPixelBufferRef` via `copyPixelBuffer` —
  a copy point, zero-copy only if IOSurface-backed** (matches this session's spike). Register on
  platform thread, consume on raster thread. (3-0) — Flutter ios-embedder docs. *(Windows/Linux
  Flutter external-texture surfaces were NOT directly verified — a gap.)*
- **Qt:** `QRhiTexture::createFrom(QRhiTexture::NativeTexture)` wraps an existing native handle
  **without taking ownership** across Vulkan (`VkImage`), Metal (`MTLTexture`), D3D12, and GL/GLES —
  a clean cross-backend embedding surface matching carapace's per-platform texture types. (3-0) —
  Qt 6 `QRhiTexture` docs.

## (5) Prior art

- **Slint + Servo** is the closest shipping reference for "embed a wgpu/GPU renderer's texture into a
  host via platform texture sharing," and the primary source for the Apple + Linux recipes above.
- The classic "skin as chrome" lineage (Winamp/WACUP, Rainmeter, conky) validates the *product*
  thesis but predates GPU texture-sharing embedding; they don't inform the cross-platform ABI.

## Biggest risks / unknowns (Windows & Linux)

1. **Everything rides `wgpu-hal` internals**, not the safe API. The interop framework (#4067) is an
   unimplemented 2023 proposal; import (#2320) has no designed API since 2021. Expect per-release
   churn; pin wgpu and track it.
2. **Linux dmabuf import was trunk-only** at research time — verify it's in the wgpu version we pin
   before committing.
3. **Windows path is indirect** (D3D11→Vulkan-backend import). Open: is there a **native DX12
   shared-texture import into wgpu's DX12 backend**, and how do WinUI / Flutter-Windows / Qt-Windows
   consume a **DXGI keyed-mutex** shared handle carapace produces?
4. **Manual lifetime + barriers.** The `DropGuard` ownership contract and explicit initial-state /
   acquire-release barriers are on us; drop-guard semantics are inconsistent across some variants.
5. **Host-surface copy hops.** Flutter-iOS forces the `CVPixelBuffer` hop (zero-copy only when
   IOSurface-backed); Windows/Linux Flutter external-texture paths are unverified.

## Open questions (carry into the ffi brainstorm)

- Safe-API timeline for wgpu external-texture interop (#4067) — do we stay on unsafe wgpu-hal
  indefinitely, and how much churn to budget?
- Native DX12 shared-texture import into wgpu's DX12 backend (vs routing through Vulkan)?
- How GTK4 / Flutter-Linux actually accept an imported dmabuf/EGLImage, incl. Wayland/DRM modifiers
  and explicit-sync / DMA-fence requirements for present pacing.
- The render-thread + command-queue design that reconciles per-host thread-affinity while minimizing
  present latency (triple-buffering / barrier scheme).
