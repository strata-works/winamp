# carapace-preview

> A live, interactive browser previewer for carapace skins. (Dev tool.)

Editing a skin used to mean a full rebuild — the native demo, or an iOS device
round-trip — just to see a change. `carapace-preview` closes that loop.

## Usage

```bash
cargo run -p carapace-preview -- path/to/skin-dir [--port <n>]
```

It serves `http://127.0.0.1:<port>` (a random free port unless `--port` is given)
and opens a browser tab. The **real carapace engine** renders the skin offscreen
(headless wgpu / Vello); the browser is a thin display + control surface.

## What you get

- **Live render** of the skin via the real engine.
- **Hot reload:** saving `skin.lua` / `skin.toml` / an asset re-renders within a
  moment. A skin that fails to load shows its **Lua error** as a banner — the
  server and last-good frame survive.
- **Click to interact:** clicking the preview forwards a pointer event, so
  `region{}` hotspots fire their actions (shown in the action log).
- **Host-data panel:** add / edit / remove the host values the skin binds
  (`value_fill{ value="level" }`, `text{ value="track" }`, …); edits re-render live.
  Removing a key (the ✕ button) drops it, so the binding falls back to its default.
- **Animated skins play** — the engine ticks continuously with wall-clock `dt`.
- **Canvas size** input re-lays-out resizable (anchored / frame) skins.

## Editing the skin

Two panels edit `skin.lua` in place, on top of the view/host-data/hot-reload
loop above:

- **Parameters panel:** lists the top-level `local NAME = <literal>` scalars
  and numeric tables (e.g. colors) found in `skin.lua`. Editing a value or a
  table subfield rewrites just that literal on disk; the file watcher picks up
  the change and re-renders.
- **Inspector:** the **Inspect** toggle switches clicks on the preview from
  pointer events to node picking. Clicking a rendered node shows its literal
  props for editing — scalar literals as a single input, and inline color/table
  literals (e.g. `color = {r,g,b,a}`) as one input per numeric subfield.
  Non-editable props show a **reason** (e.g. bound to host data, a computed
  expression, or generated by a loop) instead of a value.

Both panels edit the same way: an edit re-parses `skin.lua` with `full_moon`,
splices the exact literal span for that prop/param, and writes the file back —
it never regenerates or reformats the source. The existing hot-reload watcher
then reloads and re-renders, so the edit shows up like any other file change.

### Limitations

- **Literal-only editing.** Only props/params written as literal values are
  editable. Bound values (`value_fill{ value = "level" }`), computed
  expressions, and anything else non-literal are read-only, with a reason
  shown in the Inspector.
- **One primitive per source line.** Node-to-source correlation is line-based
  (matching the engine's runtime provenance), so multiple primitives on one
  line can't be distinguished — write skin.lua with one primitive per line to
  keep them individually pickable/editable.
- **Text nodes aren't pickable.** `text{}` nodes have no measured geometry in
  the scene graph, so the Inspector can't hit-test them yet.
- **Loop-generated nodes are read-only.** Nodes created inside a Lua loop
  report "from a loop" and can't be edited individually.

## How it works

A single engine thread owns the `Engine` (which is `!Send`), a headless wgpu
device, an offscreen `Rgba8` target, and the render loop. It renders → reads back
RGBA → PNG-encodes → pushes over a WebSocket, and only re-sends when the frame
actually changed (hash compare), so a settled static skin streams nothing. A
`tiny_http` server serves the one-page viewer; a `tungstenite` WebSocket carries
frames down and pointer/value/canvas edits up. Nothing `!Send` crosses a thread.
